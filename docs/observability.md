# Observability, metrics and Grafana

This document describes the current observability setup for the portfolio web
product after the logging, metrics, local Grafana and Grafana Cloud work.

## Scope

The observability layer is intentionally practical and low-cost:

- structured JSON logs for backend request and application events;
- request correlation through `request_id` and `X-Request-ID`;
- Prometheus-compatible HTTP and domain metrics;
- protected `/internal/metrics` endpoint;
- local Prometheus/Grafana lab for development;
- Grafana Cloud scrape of the protected production metrics endpoint;
- privacy-safe aggregate product metrics.

OpenTelemetry, traces, Loki and log shipping are intentionally out of scope for
this stage.

## Logging foundation

The backend uses structured logging to make request behaviour easier to debug
locally and in hosted environments.

Log records include stable operational fields such as:

- timestamp;
- level;
- service;
- service version;
- environment;
- logger name;
- request ID, when available;
- request route, status class and duration for HTTP request logs.

Request correlation is handled through request-context middleware. If the
visitor sends a valid `X-Request-ID`, it is reused. Otherwise, the backend
generates one and returns it in the response header.

Logs should not contain query strings, personal messages, tokens or secret-like
values.

## Metrics foundation

The backend exposes Prometheus-compatible metrics only through a protected
internal endpoint:

```text
GET /internal/metrics
Authorization: Bearer <METRICS_TOKEN>
```

Metrics are disabled by default:

```env
METRICS_ENABLED="false"
METRICS_TOKEN=""
METRICS_PATH="/internal/metrics"
```

Production must use a long random token. The local development token must not be
used outside local development.

## Metric groups

### HTTP metrics

Generated by the FastAPI instrumentation layer:

- request count;
- request duration histogram;
- request and response size metrics;
- route/handler labels;
- method and status labels.

Route labels must stay templated and low-cardinality. Raw dynamic values,
query strings and private values must not appear in metric labels.

### Domain metrics

The domain metrics cover the behaviour of the actual product:

- `portfolio_chat_requests_total`;
- `portfolio_chat_responses_total`;
- `portfolio_chat_policy_decisions_total`;
- `portfolio_rag_retrievals_total`;
- `portfolio_rag_retrieval_duration_seconds`;
- `portfolio_rag_retrieved_chunks`;
- `portfolio_llm_requests_total`;
- `portfolio_llm_request_duration_seconds`;
- `portfolio_contact_submissions_total`;
- `portfolio_escalation_events_total`;
- `portfolio_rate_limit_checks_total`.

These metrics are meant for dashboards and alerts, not for storing private
conversation content.

### Privacy-safe product metrics

The product metrics are aggregate-only:

- `portfolio_page_views_total{page="..."}`;
- `portfolio_resume_downloads_total{source="resume_page"}`.

The frontend sends only whitelisted event values. The backend validates them
before incrementing metrics.

The project does not collect:

- visitor IDs;
- user IDs;
- cookies for analytics;
- localStorage tracking IDs;
- IP hashes;
- User-Agent hashes;
- raw referrers;
- query strings;
- per-user chat usage.

## Local observability lab

The local lab is stored under:

```text
infra/observability/
```

It contains:

- Prometheus Docker Compose service;
- Grafana OSS Docker Compose service;
- Prometheus scrape config;
- Grafana datasource provisioning;
- Grafana dashboard JSON.

Useful commands:

```bash
task obs:config
task obs:up
task obs:logs
task obs:restart
task obs:down
```

Local access:

```text
Prometheus: http://localhost:9090
Grafana:    http://localhost:3001
```

Local backend metrics need:

```env
METRICS_ENABLED="true"
METRICS_TOKEN="local-dev-metrics-token"
METRICS_PATH="/internal/metrics"
```

## Grafana Cloud setup

Production monitoring uses Grafana Cloud Metrics Endpoint scraping:

```text
Render backend /internal/metrics
  <- Grafana Cloud Metrics Endpoint scrape
Grafana Cloud dashboards
```

Production Render environment variables:

```env
METRICS_ENABLED="true"
METRICS_PATH="/internal/metrics"
METRICS_TOKEN="<long-random-secret>"
```

Recommended Grafana Cloud scrape settings for low traffic:

```text
Job name: portfolio-backend
URL: https://<backend-host>/internal/metrics
Authentication: Bearer token
Scrape interval: 300s
```

Do not store `METRICS_TOKEN` in GitHub, dashboard JSON or documentation.

## Dashboard

The dashboard is stored at:

```text
infra/observability/grafana/dashboards/portfolio-observability.json
```

It is tuned for low traffic:

- default time range: last 14 days;
- slower refresh interval;
- weekly summary stat panels;
- daily count charts instead of tiny per-second rates;
- low-scale Y axes for rare events such as CV downloads.

Main dashboard sections:

- backend scrape status;
- HTTP requests and HTTP 5xx;
- HTTP latency;
- chat requests and chat policy decisions;
- RAG retrieval latency;
- LLM request outcomes;
- contact and escalation events;
- rate-limit checks;
- page views;
- resume downloads.

## Suggested alert candidates

This stage documents alert candidates but does not require them to be committed
as code.

Recommended first alerts:

1. Backend metrics scrape is down.
2. HTTP 5xx errors detected.
3. LLM/RAG failures detected.
4. Excessive rate-limit events, if spam becomes visible.

Keep alerts few and actionable. A low-traffic personal portfolio does not need a
large alerting setup.

## Operational checks

Before deployment:

```bash
task ci
```

Before changing observability files:

```bash
task obs:config
```

Manual production checks:

```bash
curl https://<backend-host>/internal/metrics
```

Expected without token: forbidden.

```bash
curl https://<backend-host>/internal/metrics \
  -H "Authorization: Bearer <METRICS_TOKEN>"
```

Expected with token: Prometheus text output.
